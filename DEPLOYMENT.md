# Deployment Guide

AI-Code-Sandbox is three services with one hard constraint: **the Sandbox Runner needs a real Docker daemon** to spawn isolated containers. That shapes where each piece runs.

```
Browser ──► Web (Vercel) ──► Orchestrator (Render) ──► Sandbox Runner (Docker host) ──► ephemeral containers
```

## 1. Sandbox Runner → a Docker-capable host

Managed container platforms (including Render's standard services) don't allow privileged docker-in-docker, so the runner must live on a host with a Docker daemon: a small cloud VM (DigitalOcean, EC2, Hetzner), a Fly.io machine, or similar.

On that host:

```bash
git clone <repo> && cd AI-Code-Sandbox
# Build the language images into the host daemon:
bash infra/build-images.sh
# Build and run the runner with access to the Docker socket:
docker build -t ai-sandbox-runner ./sandbox
docker run -d --name sandbox-runner \
  -p 8090:8090 \
  -v /var/run/docker.sock:/var/run/docker.sock \
  -e MAX_MEMORY_MB=512 -e MAX_CONCURRENCY=8 \
  ai-sandbox-runner
```

Keep port 8090 private — expose it only to the orchestrator (security group / private network / firewall). The runner has no auth of its own by design; it is an internal service.

> Hardening note: mounting the Docker socket grants the runner control of the host daemon. Run it on a dedicated host used only for sandboxing.

## 2. Orchestrator → Render

The repo ships `render.yaml`. In Render: **New → Blueprint**, point it at this repo. Then set:

- `SANDBOX_URL` → your runner URL (e.g. `http://<runner-host>:8090` over a private network, or an HTTPS tunnel).
- `ALLOWED_ORIGINS` → your Vercel URL (e.g. `https://your-app.vercel.app`).
- `API_KEY` → auto-generated by the blueprint; copy it for the frontend.

Health check is `/health`.

## 3. Web → Vercel

In Vercel: **New Project → import the repo**, set **Root Directory = `web`**. Vercel auto-detects Next.js. Add environment variables:

- `NEXT_PUBLIC_API_BASE_URL` → your Render orchestrator URL (e.g. `https://ai-code-sandbox-orchestrator.onrender.com`).
- `NEXT_PUBLIC_API_KEY` → the `API_KEY` from Render (optional; only if auth is enabled).

For the AI code-generation composer (server-side only — **do not** prefix these with `NEXT_PUBLIC`, or you'll leak the key to the browser):

- `OPENROUTER_API_KEY` → your OpenRouter key (from https://openrouter.ai/keys).
- `OPENROUTER_MODELS` → comma-separated free models, e.g. `qwen/qwen3-coder:free,meta-llama/llama-3.3-70b-instruct:free`.

The `NEXT_PUBLIC_*` vars are inlined at build time, so redeploy after changing them. `OPENROUTER_*` are read at request time by the `/api/generate` route.

## Local (everything at once)

```bash
docker compose up --build
# Web:  http://localhost:3000
# API:  http://localhost:8000
```

Compose builds the language images, wires the runner to your local Docker socket, and starts all three services. Set `API_KEY` in a root `.env` to test authenticated mode.

## Configuration reference

| Service | Key vars |
|---|---|
| Runner | `MAX_MEMORY_MB`, `MAX_CPUS`, `MAX_PIDS`, `MAX_TIMEOUT_MS`, `MAX_CONCURRENCY` |
| Orchestrator | `SANDBOX_URL`, `API_KEY`, `ALLOWED_ORIGINS`, `RATE_LIMIT_PER_MINUTE` |
| Web | `NEXT_PUBLIC_API_BASE_URL`, `NEXT_PUBLIC_API_KEY`, `OPENROUTER_API_KEY`, `OPENROUTER_MODELS` |
